整体架构方案

  基于你的需求文档，我设计以下模块化方案（每层独立文件夹）：

  目录结构

  auto_agents_build/
  ├── role_alignment/          # 第二层：职能对齐层
  ├── org_fusion/              # 第三层：组织融合层
  ├── artifact_generation/     # 第四层：代码生成层
  ├── shared/                         # 共享组件
  └── orchestrator/                   # 总控编排器

  Layer 2: 职能对齐层 (layer2_role_alignment/)

  核心职责

  将技术性 API 能力映射到业务性职能角色

  模块设计

  1. functional_meta_library.py - 全领域职能元库
  - 预置多领域标准角色库（CRM/MES/财务/HR等）
  - 支持 JSON 格式存储和向量化索引
  - 提供领域模板的 CRUD 操作

  2. domain_detector.py - 领域探测器
  - 分析 Layer I 的 API 聚类结果
  - 使用 LLM 计算领域权重（关键词统计 + 语义分析）
  - 输出主导领域和次要领域列表

  3. semantic_alignment_engine.py - 语义对齐引擎
  - 阶段1: 向量粗筛（Embedding + 余弦相似度）
  - 阶段2: LLM 终审（基于业务逻辑推理）
  - 处理一对多、多对一的映射关系

  4. template_loader.py - 模板加载器
  - 根据领域探测结果加载"黄金模板"
  - 支持模板合并（多领域场景）
  - 无模板时触发 LLM 生成新模板

  5. capability_slotter.py - 能力填装器
  - 将原子能力自动填入模板槽位
  - 实现 Required/Optional 能力的优先级匹配
  - 记录未匹配的"孤儿 API"

  6. gap_analyzer.py - 差异分析器
  - 角色瘦身: 冻结无 API 支撑的空角色
  - 虚拟角色生成: 为孤儿 API 聚类创建新角色
  - 输出角色调整报告

  7. constraint_injector.py - 约束注入器
  - 为敏感 API 添加前置审计契约
  - 注入数据隔离规则（如 owner_id 过滤）
  - 生成业务契约清单

  8. role_manifest_generator.py - 清单生成器
  - 输出《职能装配清单》（YAML/JSON）
  - 包含角色描述、分配的 API、约束规则

  输入/输出

  - 输入: Layer I 的原子能力 JSON
  - 输出: role_assembly_manifest.json

  Layer 3: 组织融合层 (layer3_org_fusion/) - 组合模式重构

  核心理念转变

  从"树形继承"到"能力组合"：
  - 底层能力是独立的"能力单元"（Capability Unit）
  - 上层 Agent 通过"组合引用"而非"拥有"来使用能力
  - 多个 Agent 可以同时引用同一个能力单元
  - 类似 Go 的 interface 组合和 struct embedding

  ---
  重构后的模块设计

  1. capability_unit_registry.py - 能力单元注册中心

  职责：
  - 将所有底层 API 和中层能力抽象为独立的"能力单元"
  - 每个能力单元有唯一 ID，可被多个 Agent 引用
  - 维护能力单元的元数据和依赖关系

  3. capability_composer.py - 能力组合器（新增核心模块）

  职责：
  - 根据 Agent 的职责，智能组合底层能力单元
  - 处理能力的"横向复用"和"纵向晋升"
  - 生成能力的组合逻辑（类似 Go 的方法集）

  组合策略：

  A. 横向复用（Horizontal Reuse）
  场景：多个同级 Agent 需要相同能力

  订单专员 ──┐
             ├──> [cap_order_create] <── 共享引用
  销售主管 ──┘

  策略：
  - 能力单元保持单例
  - 每个 Agent 持有引用指针
  - 通过 access_mode 控制权限（execute/read-only）

  B. 纵向晋升（Vertical Promotion）
  场景：上级需要下级能力的抽象视图

  仓库主管 ──> [cap_inventory_mgmt] (composed)
                ├─ cap_query_stock
                ├─ cap_update_stock
                └─ cap_alert_setup
                     ↑
                     └── 仓库专员直接使用原子能力

  策略：
  - 创建新的 composed 能力单元
  - 包含对底层能力的引用
  - 上级通过 composed 能力调用，下级直接调用原子能力

  C. 跨域组合（Cross-Domain Composition）
  场景：高层 Agent 需要跨部门协调

  CEO ──> [cap_strategic_decision] (strategic)
           ├─ cap_sales_pipeline (销售域)
           ├─ cap_inventory_mgmt (仓储域)
           └─ cap_financial_report (财务域)

  策略：
  - 创建 strategic 级别的能力单元
  - 包含跨域能力的编排逻辑
  - 通过 delegate_to 指定执行者

  4. capability_promoter.py - 能力晋升器（重构）

  职责：
  - 将多个原子能力晋升为组合能力
  - 使用 LLM 生成语义化的能力描述
  - 定义能力间的编排逻辑

  晋升算法：
  def promote_capabilities(atomic_caps, target_agent_level):
      """
      输入：原子能力列表 + 目标 Agent 层级
      输出：晋升后的组合能力
      """
      # 1. 语义聚类：将相关的原子能力分组
      clusters = semantic_cluster(atomic_caps)

      # 2. LLM 生成抽象描述
      for cluster in clusters:
          composed_cap = {
              "name": llm_generate_name(cluster),
              "description": llm_generate_description(cluster),
              "composed_of": [cap.unit_id for cap in cluster],
              "orchestration": llm_generate_orchestration(cluster)
          }

      # 3. 注册到能力单元中心
      registry.register(composed_cap, level="composed")

      return composed_cap

  示例：
  输入：
  - cap_query_stock (查询库存)
  - cap_update_stock (更新库存)
  - cap_alert_setup (设置预警)

  LLM 分析：
  "这三个能力都围绕库存管理，可以组合为一个高阶能力"

  输出：
  {
    "unit_id": "cap_inventory_health",
    "name": "库存健康度管理",
    "level": "composed",
    "composed_of": ["cap_query_stock", "cap_update_stock", "cap_alert_setup"],
    "orchestration_logic": "定期查询库存状态，根据阈值更新库存，触发预警机制",
    "semantic_interface": "能够全面管理库存的健康状态"
  }

  5. topology_builder.py - 拓扑结构构建器（重构）

  职责：
  - 构建基于 Neo4j 的图结构
  - 节点类型：Agent 节点 + Capability 节点
  - 关系类型：MANAGES（管理）、REFERENCES（引用）、COMPOSES（组合）

  图结构设计：
  // Agent 节点
  (:Agent {
    id: "agent_sales_supervisor",
    name: "销售主管",
    level: "supervisor"
  })

  // Capability 节点
  (:Capability {
    id: "cap_order_create",
    name: "订单创建能力",
    level: "atomic"
  })

  // 关系
  (销售主管)-[:REFERENCES {access_mode: "execute"}]->(订单创建能力)
  (订单专员)-[:REFERENCES {access_mode: "execute"}]->(订单创建能力)
  (销售主管)-[:MANAGES]->(订单专员)
  (库存健康度管理)-[:COMPOSES]->(查询库存)

  查询能力复用：
  // 查找被多个 Agent 引用的能力
  MATCH (a:Agent)-[:REFERENCES]->(c:Capability)
  WITH c, count(a) as ref_count, collect(a.name) as agents
  WHERE ref_count > 1
  RETURN c.name, ref_count, agents

  6. supervisor_synthesizer.py - 主管合成器（重构）

  职责：
  - 根据拓扑结构动态生成主管 Agent
  - 主管不"拥有"下属的能力，而是"引用"或"委托"
  - 生成主管的"能力视图"（View）

  主管能力视图生成：
  def synthesize_supervisor_view(supervisor_id, subordinates):
      """
      为主管生成能力视图
      """
      view = {
          "direct_capabilities": [],      # 主管自己可执行的
          "delegated_capabilities": [],   # 需要委托给下属的
          "monitoring_capabilities": []   # 只监控不执行的
      }

      # 1. 收集下属的所有能力引用
      all_sub_caps = []
      for sub in subordinates:
          all_sub_caps.extend(sub.capability_interface.direct_capabilities)

      # 2. 晋升为组合能力
      composed_caps = capability_promoter.promote(all_sub_caps, level="supervisor")

      # 3. 分类能力
      for cap in composed_caps:
          if cap.requires_coordination:
              view["direct_capabilities"].append(cap)  # 需要主管协调
          elif cap.is_routine:
              view["delegated_capabilities"].append(cap)  # 常规任务委托
          else:
              view["monitoring_capabilities"].append(cap)  # 仅监控

      return view

  7. capability_access_controller.py - 能力访问控制器（新增）

  职责：
  - 管理能力的访问权限
  - 处理能力的并发访问（多个 Agent 同时调用）
  - 实现能力的"读写锁"机制

  访问模式：
  class AccessMode(Enum):
      EXECUTE = "execute"      # 可执行（读写）
      READ_ONLY = "read_only"  # 只读
      DELEGATE = "delegate"    # 委托执行
      MONITOR = "monitor"      # 仅监控

  并发控制：
  def acquire_capability(agent_id, capability_id, mode):
      """
      获取能力的访问权限
      """
      cap = registry.get(capability_id)

      # 检查访问权限
      if not has_permission(agent_id, capability_id, mode):
          raise PermissionDenied()

      # 并发控制
      if mode == AccessMode.EXECUTE:
          if cap.is_locked():
              return wait_or_queue(agent_id, capability_id)
          cap.lock(agent_id)

      return CapabilityHandle(cap, mode)

  关键优势

  1. 真正的能力复用：
    - 一个能力单元可被多个 Agent 引用
    - 避免能力的重复定义和维护
  2. 灵活的组合：
    - 支持横向复用（同级共享）
    - 支持纵向晋升（层级抽象）
    - 支持跨域组合（战略协调）
  3. 清晰的职责边界：
    - 通过 access_mode 明确权限
    - 通过 contract 定义使用规则
    - 通过 lender/borrower 明确责任人
  4. 动态可扩展：
    - 新增 Agent 只需引用现有能力
    - 新增能力自动对所有 Agent 可见
    - 支持运行时的能力装卸载
  5. 类似 Go 的组合哲学：
    - "组合优于继承"
    - 接口隔离（每个 Agent 只看到需要的能力）
    - 显式依赖（通过引用而非隐式继承）

  数据流示例（组合模式）

  场景：销售总监需要协调订单和库存

  1. Layer 2 输出（能力单元）：
     - cap_order_create (订单专员负责)
     - cap_inventory_query (仓库专员负责)

  2. Layer 3 处理：

     a) 注册能力单元：
        registry.register(cap_order_create, owner="订单专员")
        registry.register(cap_inventory_query, owner="仓库专员")

     b) 销售总监引用能力：
        销售总监.capability_interface = {
          "direct_refs": ["cap_order_create"],
          "delegated_refs": ["cap_inventory_query"]
        }

     c) 生成组合能力：
        cap_sales_coordination = compose([
          cap_order_create,
          cap_inventory_query
        ], orchestration="先查库存，再创建订单")

     d) 建立契约：
        contract_001: 销售总监 借用 cap_order_create (execute)
        contract_002: 销售总监 借用 cap_inventory_query (read_only)

  3. Layer 3 输出：
     {
       "agent": "销售总监",
       "capability_refs": ["cap_order_create", "cap_inventory_query"],
       "composed_capability": "cap_sales_coordination",
       "contracts": ["contract_001", "contract_002"]
     }

  完整工作流程

  1. 用户输入
     ├─ Swagger API 文档
     ├─ 组织架构表（可选）
     └─ 配置参数

  2. Orchestrator 启动
     └─ 验证输入 → 初始化各层模块

  3. Layer 2 执行
     ├─ 领域探测 → 加载模板
     ├─ 语义对齐 → 能力填装
     ├─ 差异分析 → 约束注入
     └─ 输出: role_assembly_manifest.json

  4. Layer 3 执行
     ├─ Agent 封装 → 拓扑构建
     ├─ 主管合成 → 能力晋升
     ├─ 契约管理 → 冲突解决
     └─ 输出: org_runtime_blueprint.json

  5. Layer 4 执行
     ├─ Prompt 生成 → 配置生成
     ├─ → RAG 链接

  - 优化: 批量调用、结果缓存、模型降级（简单任务用小模型）

  2. 向量化策略

  - API 描述 Embedding（用于相似度匹配）
  - 角色职责 Embedding（用于检索）
  - 使用维度: 768 或 1536（根据模型选择）

  3. 容错机制

  - 每层输出都有 Schema 验证
  - 关键步骤支持人工介入（如模糊匹配需确认）
  - 失败时保存中间结果，支持断点续传

  4. 可扩展性

  - 插件化的领域模板（用户可自定义）
  - 可替换的 LLM 后端
  - 模块化设计，每层可独立升级
